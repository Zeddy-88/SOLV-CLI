# Design Docs — Core Beliefs

이 문서는 제품을 만드는 모든 결정의 기준이 되는 설계 원칙.  
기능을 추가하거나 UX를 결정할 때 "이 원칙에 맞는가?"를 먼저 확인한다.

---

## 1. 영업사원이 주인공이다

이 앱의 사용자는 재무 전문가가 아니다.  
재무제표를 처음 보는 영업사원도 5분 안에 핵심을 파악하고 대표 앞에 앉을 수 있어야 한다.

**실천:**
- 재무 용어는 반드시 해석과 함께 제공한다. 수치만 나열하지 않는다.
- "이자보상배수 1.93배"가 아니라 "이자를 겨우 갚을 수 있는 수준"으로 풀어준다.
- 분석 결과 화면에서 영업사원이 "그래서 내가 뭘 말하면 돼?"에 바로 답할 수 있어야 한다.

---

## 2. 결과물은 두 가지 독자를 위한 것이다

**영업사원용 브리핑 화면** — 미팅 전 준비 자료. 이슈, 수치 근거, 영업 멘트까지 모두 포함.  
**대표 제출용 PDF** — 미팅 자리에서 건네는 자료. 문제와 가능성만. 영업 멘트는 절대 포함하지 않는다.

두 결과물의 독자와 목적이 다르다는 사실을 항상 기억한다.  
같은 내용을 두 곳에 다 넣으려는 충동을 경계한다.

---

## 3. 단순함이 신뢰를 만든다

정보가 많을수록 좋은 게 아니다. 재무 데이터는 원래 방대하다.  
우리가 할 일은 그 중에서 **지금 이 대표에게 중요한 것 3가지**를 골라내는 것이다.

**실천:**
- 이슈는 최대 5개(브리핑), 3개(PDF)로 제한한다.
- 전체 재무비율 목록을 그대로 나열하지 않는다.
- 숫자는 항상 맥락과 함께 제시한다. "영업이익률 1.27%"가 아니라 "업종 평균(4.2%)의 1/3 수준"으로.

---

## 4. 영업은 설득이 아니라 공감에서 시작한다

AI가 생성하는 영업 멘트는 대표를 압박하거나 공포심을 조장하는 방향이 되어서는 안 된다.  
대표가 이미 알고 있지만 말로 꺼내지 못한 것을 대신 꺼내주는 방식이어야 한다.

**실천:**
- 멘트는 항상 "공감 → 문제 제시 → 위험 인식 → 가능성 제안" 순서를 따른다.
- "지금 당장 계약하지 않으면 망합니다" 식의 표현은 생성하지 않는다.
- 강점 기반의 클로징을 반드시 포함한다. 문제만 나열하고 끝나지 않는다.

---

## 5. 데이터가 없으면 없다고 말한다

PDF 포맷에 따라 제공되는 데이터가 다르다.  
재무진단 등급이 없는 PDF라면 그 섹션을 빈칸으로 두거나 "데이터 없음"을 표시하지 않는다.  
그냥 숨긴다.

없는 데이터를 억지로 채우거나, AI가 추정으로 생성하거나, 빈 공간을 어색하게 남기지 않는다.

**실천:**
- 조건부 렌더링을 적극 활용한다. 데이터가 없는 섹션은 렌더링하지 않는다.
- AI 분석 시 데이터가 부족한 항목은 해당 필드를 `null`로 반환하도록 프롬프트에 명시한다.
- `null` 처리 로직은 컴포넌트 레벨이 아닌 데이터 레이어에서 처리한다.

---

## 6. 속도가 신뢰다

영업사원은 미팅 직전에 이 앱을 열 수도 있다.  
"아직 분석 중입니다"가 3분 넘게 이어지면 앱을 닫는다.

**실천:**
- 분석 진행 상태를 실시간으로 보여준다. 침묵하지 않는다.
- PDF 업로드 → 분석 완료까지 목표 시간: 60초 이내.
- 느리더라도 진행 중임을 알리는 피드백이 없는 것보다 낫다.

---

## 7. 보안은 타협하지 않는다

영업사원이 올리는 PDF에는 기업의 민감한 재무 정보가 담겨 있다.  
사용자는 자신이 업로드한 데이터만 볼 수 있어야 한다. 예외 없이.

**실천:**
- 모든 DB 테이블에 RLS를 적용한다. 없으면 배포하지 않는다.
- API Route 진입 시 항상 세션을 검증한다.
- 서비스 키는 클라이언트에 절대 노출하지 않는다.

---

## 이 원칙을 어길 때

기능 개발 중 이 원칙과 충돌하는 결정이 필요하다면,  
`docs/exec-plans/` 또는 PR 설명에 이유를 명시하고 의식적으로 어긴다.  
모르는 채로 어기는 것과 알면서 어기는 것은 다르다.
