# RELIABILITY.md — 안정성 가이드

이 앱은 영업사원이 미팅 직전에 여는 도구다.  
"지금 안 되면 미팅을 망친다"는 상황이 발생하지 않도록 안정성을 설계한다.

---

## 핵심 원칙

**침묵하지 않는다.** 무슨 일이 일어나고 있는지 항상 사용자에게 알린다.  
**부분 실패를 허용한다.** 한 섹션이 깨져도 다른 섹션은 정상 표시되어야 한다.  
**복구 경로를 제공한다.** 오류가 발생하면 항상 "다음에 할 수 있는 것"을 안내한다.

---

## API 및 외부 서비스 장애 대응

### Claude API 호출 실패

| 상황 | 처리 |
|---|---|
| 타임아웃 (60초 초과) | 자동 재시도 1회 → 실패 시 `status: error` 저장 |
| JSON 파싱 실패 | 재시도 1회 → 실패 시 오류 처리 |
| API 키 오류 | 재시도 없이 즉시 오류 처리 |
| 응답 불완전 (필드 누락) | 누락 필드를 `null`로 처리하고 나머지는 표시 |

재시도 로직은 `lib/claude/retry.ts`에서 관리.  
최대 재시도 횟수: 2회. 그 이상은 사용자에게 오류 안내.

### Supabase 장애

| 상황 | 처리 |
|---|---|
| DB 조회 실패 | 오류 메시지 + 새로고침 안내 |
| Storage 업로드 실패 | "파일 저장 중 문제가 발생했어요. 다시 시도해주세요." |
| Realtime 연결 끊김 | 폴링 방식으로 자동 폴백 (5초 간격) |

## 에러 바운더리

### 전역 에러 바운더리 (`app/error.tsx`)

예상치 못한 런타임 오류 발생 시:
- "예상치 못한 오류가 발생했어요." 메시지
- "새로고침" 버튼
- 오류 상세 (개발 환경에서만 표시)

### 페이지별 에러 처리

- `/dashboard/[id]` — 분석 오류 시 해당 섹션만 오류 표시. 전체 페이지 크래시 없음.
- `/history` — 목록 로드 실패 시 빈 상태가 아닌 오류 메시지 표시.

## 로딩 상태 설계

로딩 중 빈 화면을 사용자에게 보여주지 않는다.

| 상황 | 처리 |
|---|---|
| 브리핑 화면 초기 로드 | 섹션별 스켈레톤 UI |
| 분석 진행 중 | 단계별 상태 텍스트 + 프로그레스 바 |
| PDF 업로드 중 | "파일 업로드 중..." 텍스트 |
| 이력 목록 로드 | 카드 스켈레톤 3~5개 |

## Vercel 서버리스 제한 대응

Vercel 함수는 기본 타임아웃 10초. Claude API 호출은 최대 60초가 필요하다.

설정 방법:
```typescript
// app/api/analyze/route.ts
export const maxDuration = 60  // Pro 플랜 필요
```

**주의:** 무료 플랜에서는 `maxDuration = 60` 설정이 불가. Pro 플랜 이상에서 배포.

## 데이터 일관성

### 분석 상태 동기화

`analyses.status` 필드가 `processing` 상태로 멈추는 경우 (서버 오류로 완료 처리가 안 된 경우):
- 10분 이상 `processing` 상태 유지 시 `error`로 자동 전환 (Supabase Edge Function 또는 cron으로 처리)
- 사용자에게 "분석이 완료되지 않았어요. 다시 시도해주세요." 안내

### 파일 정합성

- Storage에 파일이 존재하지만 DB 레코드가 없는 경우: 무시 (孤立 파일)
- DB 레코드가 있지만 Storage 파일이 없는 경우: 재분석 불가 안내

## 모니터링

MVP 단계에서는 간단한 수준으로 유지.

- Vercel 대시보드의 함수 오류 로그 확인
- Supabase 대시보드의 DB 오류 및 API 요청 로그 확인
- `console.error`로 서버사이드 오류를 명확히 기록

Post-MVP에서 추가 검토: Sentry 또는 유사 모니터링 도구 도입.

[ ] 타임아웃 및 재시도 로직 동작 확인
